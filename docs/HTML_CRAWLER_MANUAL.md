# HTML AI 爬虫使用手册

更新时间：2026-03-31

## 文档目的

本文档面向 MicroFlow 中“自定义数据源 -> 网页提取”功能，整理一套可直接执行的操作手册。

文档重点不是解释底层实现，而是帮助你按固定流程完成：

1. 添加规则
2. 调试规则
3. 保存规则
4. 复测规则
5. 排查问题

如果只想快速上手，建议至少完整走一遍“添加规则 -> 调试 -> 保存 -> 复测”。

## 当前能力范围

HTML AI 爬虫当前适合以下目标站点：

- 高校门户、学院站、通知公告站
- 政府信息公开、公示公告类页面
- 常见 CMS 新闻页、文章页、公告页
- 需要分页、需要详情页正文、需要附件和图片提取的列表页

当前已经支持：

- AI 自动分析网页结构并生成规则
- 站点画像识别与模板库推荐
- 浏览器兜底抓取
- GET / POST、自定义请求头、Cookie / 登录态
- 单页、下一页、页码模板、加载更多
- 列表页正文、详情页正文、混合正文
- 时间、附件、图片、表格、标题层级保留
- 详情预览、手动微调、一键复测
- 健康监控、规则版本、回滚、导入导出

当前已知边界：

- `closed shadow root` 无法通用抓取
- 强验证码、强风控站点不保证可自动通过
- 极重前端渲染站点可能仍需手动微调

## 使用前准备

开始前请先确认：

- 已在设置中配置可用的 AI 模型、`API Key`、`baseUrl`
- 当前目标网页在本机网络环境下可访问
- 如果网页需要登录态，已准备好浏览器里的 Cookie
- 如果网页使用接口 POST 拉列表，已准备好请求体和必要请求头

建议优先准备一个“列表页” URL，而不是文章详情页 URL。

理想入口形态：

- 新闻列表页
- 公告列表页
- 通知公告栏目页
- 带分页的文章索引页

## 一、添加规则

### 1. 进入入口

操作路径：

- 打开设置
- 进入“自定义数据源”
- 点击“添加”
- 选择 `网页提取`

### 2. 填写基础信息

先填写以下字段：

- `任务名称`
  - 建议写真实来源名，例如“创意设计学院通知公告”
- `任务目的`
  - 可写“通知公告”“新闻动态”“学术讲座”等
- `目标网页 URL`
  - 填栏目列表页，不要填单篇正文页
- `最大抓取条数`
  - 一般先填 `10`
  - 如果只是调试，可先填 `5`

### 3. 选择抓取方式

界面字段：`抓取方式`

可选值：

- `请求优先，失败后浏览器兜底`
- `浏览器优先，失败后请求兜底`
- `仅使用请求`
- `仅使用浏览器`

推荐规则：

- 普通高校站、政府站、传统 CMS：先用 `请求优先`
- 页面打开后内容明显是 JS 渲染出来的：改成 `浏览器优先`
- 明确知道站点必须浏览器跑 JS 才出列表：用 `仅使用浏览器`
- 只想测纯静态 HTML 稳定性：用 `仅使用请求`

### 4. 配置高级访问

界面字段：`高级访问`

这里主要用于三类场景：

- 列表接口是 `POST`
- 站点需要特殊请求头
- 站点需要登录态 Cookie

可填写内容：

- `请求方法`
  - `GET` 或 `POST`
- `POST 请求体`
  - 支持 JSON
  - 也支持 `key=value&key2=value2`
- `自定义请求头`
  - 每行一条，格式：`Key: Value`
- `Cookie / 登录态`
  - 直接粘贴浏览器里的原始 Cookie 字符串

适用示例：

- 校内统一门户页要求 `Referer`
- 接口页需要 `X-Requested-With`
- 登录后才能看到通知列表

### 5. 配置分页

界面字段：`列表分页`

分页模式说明：

- `单页抓取`
  - 适合最新消息都在第一页的栏目
- `下一页链接`
  - 适合页底有“下一页”的站点
- `页码模板`
  - 适合 URL 明显带页码模板的站点
  - 例如 `/list_{page}.htm`
- `加载更多`
  - 适合按钮点击后继续加载列表的站点

推荐填写：

- `最大翻页数`
  - 初次调试建议 `2` 或 `3`
- `常规追踪最大翻页数`
  - 正式运行建议 `1`
  - 避免每次轮询都翻太多旧页

经验建议：

- 如果站点每天更新频率低，`常规追踪最大翻页数=1` 一般足够
- 如果站点经常把新内容插在第二页，调到 `2`

### 6. 配置提取字段

界面字段：`提取字段`

推荐最小字段集：

- `title`
- `url`
- `date`

可按需要追加：

- `summary`
- `author`
- `category`
- `department`

建议：

- 第一次不要贪多
- 先让 `title / url / date` 稳定命中
- 其他字段等主链路稳定后再加

### 7. 配置正文策略

界面字段：

- `列表正文字段`
- `正文抓取策略`
- `详情提取规则`

正文抓取策略说明：

- `详情页优先`
  - 最通用
  - 正文优先抓详情页，失败回退列表
- `混合模式`
  - 详情正文和列表字段一起参与拼接
  - 适合列表有摘要、详情有正文的站点
- `列表页足够`
  - 不抓详情
  - 适合列表页已经带完整摘要或正文

推荐：

- 普通新闻站：`详情页优先`
- 列表页有摘要、详情页正文不稳定：`混合模式`
- API 直接返回完整正文：`列表页足够`

详情提取规则可选项：

- `正文容器选择器`
- `时间选择器`
- `附件区域选择器`
- `图片区选择器`

第一次使用时可以先留空，让 AI 自动分析；当详情预览不稳定时，再手动补这些选择器。

## 二、调试规则

### 1. 先点 AI 分析

在基础信息填写完成后，点击：

- `AI 智能分析并测试`

系统会做以下事情：

1. 拉取目标网页
2. 判断站点画像
3. 尝试匹配模板库
4. 如果模板命中，则优先试模板
5. 如果模板不够，就调用 AI 生成规则
6. 生成列表样本预览
7. 生成详情样本预览
8. 返回页面摘要和测试快照

### 2. 看预览区时重点看什么

预览区里重点关注 4 块：

- `站点画像`
  - 系统判断这个站点更像高校站、政府站还是 CMS 站
- `命中模板 / 推荐模板`
  - 如果命中模板，说明系统已经找到较合适的结构参考
- `列表样本`
  - 看 `title / url / date` 是否都抽对
- `详情预览`
  - 看正文、时间、附件、图片是否命中

判断标准：

- 列表样本至少要抽到正确标题和链接
- 日期最好正确
- 详情预览如果要求通过，就不能是空正文

### 3. 如何判断当前问题出在哪

现象与判断：

- 列表样本一条都没有
  - 多半是列表容器或列表项没抽对
  - 也可能网页需要浏览器渲染
- 列表样本有标题，但链接是空的
  - `url` 字段选择器不对
- 列表样本都对，但详情预览失败
  - 详情正文容器没命中
  - 或详情页结构与列表页结构完全不同
- 详情有正文，但时间不对
  - 详情时间选择器需手动补
- 附件和图片没有
  - 详情附件 / 图片区域选择器需补

### 4. 什么时候需要手动微调

出现以下情况时，建议打开 `手动微调选择器 (Pro)`：

- AI 生成的列表容器太宽或太窄
- 样本里出现重复项
- 抽到了导航、侧边栏、推荐区
- 标题对了，但日期总为空
- 明显抽到了错误栏目

优先微调顺序：

1. `列表容器`
2. `列表项选择器`
3. `field_selectors`
4. 分页选择器
5. 详情选择器

每次修改后都要点击：

- `用此规则重新测试`

不要连续改很多项再一起测，否则很难定位问题来源。

### 5. 常见调试策略

#### 场景 A：页面打开后列表才出现

做法：

- 抓取方式切到 `浏览器优先`
- 重新测试

#### 场景 B：接口是 POST 拉列表

做法：

- `请求方法` 设为 `POST`
- 粘贴请求体
- 必要时补 `Referer`、`Content-Type`
- 重新测试

#### 场景 C：登录后才能看

做法：

- 粘贴浏览器 Cookie
- 必要时加请求头
- 优先试 `请求优先`
- 如果页面依赖前端登录态渲染，再试 `浏览器优先`

#### 场景 D：详情页正文老是空

做法：

- 先把 `正文抓取策略` 设为 `详情页优先`
- 手动补 `正文容器选择器`
- 如果详情页有正文，列表页也有摘要，改成 `混合模式`
- 如果列表页已经足够，直接改成 `列表页足够`

#### 场景 E：分页站点第一页正常，第二页抓不到

做法：

- 检查分页模式是否选错
- `下一页链接` 是否应该写成 `a.next::attr(href)`
- `页码模板` 是否用了 `{page}`
- `最大翻页数` 是否太小

## 三、保存规则

### 1. 保存前检查清单

建议在点击保存前确认以下项目：

- 列表样本至少有 1 到 3 条看起来正确
- `title` 正确
- `url` 正确且可打开
- `date` 尽量正确
- 如果不是 `列表页足够`，详情预览最好通过
- 如果页面有附件或图片，至少抽到一部分
- 没有把导航、推荐文章、侧边栏误抽进来

### 2. 保存动作

确认无误后点击：

- `确认无误，保存规则`

保存后会发生：

- 规则写入本地规则存储
- 生成一个可回滚的版本历史
- 调度器热加载新规则
- 后续轮询会正式使用这条规则

### 3. 保存后的版本能力

HTML 规则保存后具备：

- 历史版本
- 回滚
- 导入 / 导出
- 最近一次测试快照
- 页面摘要

所以如果后续调坏了，不一定要从零重建。

## 四、复测规则

### 1. 什么时候要复测

建议在以下场景执行“一键复测”：

- 规则刚保存完，想确认运行态没问题
- 网站最近改版了
- 健康状态出现异常
- 文章明显抓少了
- 抓到的数据结构变了

### 2. 如何复测

在已保存规则上执行：

- `一键复测`

这和编辑态的“测试规则”不同：

- 编辑态测试：重点看预览效果
- 一键复测：走接近正式运行的抓取链路，并回写健康状态

### 3. 如何理解健康状态

你在规则列表中可能会看到这些状态：

- `运行正常`
- `字段异常`
- `暂无新内容`
- `持续空抓`
- `网络异常`
- `选择器失效`
- `列表容器失效`
- `列表项失效`

建议处理方式：

- `运行正常`
  - 不用动
- `暂无新内容`
  - 站点可能只是这次没更新
- `持续空抓`
  - 优先复测，再看列表结构是否改版
- `字段异常`
  - 说明某些字段命中率下降，建议重新测试
- `列表容器失效 / 列表项失效`
  - 大概率结构改版，建议重生成或手调
- `网络异常`
  - 优先排查站点可访问性、请求头、Cookie、代理环境

### 4. 什么时候该回滚

适合回滚的情况：

- 新保存的版本明显比旧版本更差
- 站点临时异常，AI 重生成反而拟合错了
- 旧版本还能抓，新版本完全抓不到

适合重生成的情况：

- 页面结构已经明显改版
- 旧规则持续空抓
- 列表容器或字段选择器整体漂移

## 五、排障

### 1. 问题：AI 分析后没有列表样本

优先排查：

- URL 是否是列表页而不是正文页
- 页面是否需要浏览器渲染
- 列表是否在 iframe 中
- 是否需要登录态

处理建议：

1. 切到 `浏览器优先`
2. 如果需要，补 Cookie
3. 如果是 POST 列表，补请求体
4. 重新测试

### 2. 问题：列表样本对了，但详情正文为空

原因通常有：

- 详情正文容器没命中
- 正文在 iframe 中
- 正文是延迟渲染
- 详情页没有正文，只有列表摘要

处理建议：

1. 手动填 `正文容器选择器`
2. 如果详情页结构特殊，补 `时间 / 附件 / 图片` 选择器
3. 如果列表摘要已经够用，改成 `列表页足够`
4. 如果列表有摘要、详情有正文，改成 `混合模式`

### 3. 问题：分页只抓第一页

优先排查：

- 分页模式是否选错
- 下一页链接是不是相对链接
- 页码模板是不是没写 `{page}`
- 加载更多按钮是不是点一次后 URL 不变

处理建议：

1. 重新确认分页模式
2. 对 `next_link` 使用 `::attr(href)`
3. 对 `url_template` 使用明确模板
4. 对 `load_more` 检查按钮选择器是否稳定

### 4. 问题：网站需要登录才能看

处理建议：

1. 登录网页
2. 从浏览器复制 Cookie
3. 粘贴到 `Cookie / 登录态`
4. 如果接口校验来源，再补 `Referer`
5. 重新测试

如果仍失败：

- 优先尝试 `浏览器优先`

### 5. 问题：总是抽到导航栏、推荐区、侧边栏

原因通常是：

- `列表容器` 太宽
- `item_selector` 太泛

处理建议：

1. 手动缩小 `列表容器`
2. 优先用更语义化的选择器
3. 避免把整个 `main` 或整个 `body` 当列表容器

### 6. 问题：链接对了，但日期总为空

处理建议：

1. 先确认列表项里是否真的有日期
2. 如果日期只在详情页，补 `详情时间选择器`
3. 如果字段名不是 `date`，统一成 `date` 更利于模板和调试

### 7. 问题：浏览器抓取超时

可能原因：

- 站点加载太慢
- 站点有复杂前端渲染
- 页面被风控或脚本拦截

处理建议：

1. 先试 `请求优先`
2. 只在必要时用 `浏览器优先`
3. 如果浏览器也不稳定，考虑换更稳定的栏目入口页

### 8. 问题：规则之前正常，现在突然异常

说明很可能发生了结构漂移。

建议顺序：

1. 先看健康状态
2. 执行一键复测
3. 如果只是小漂移，手动微调
4. 如果整体结构变了，重新生成
5. 新版本不如旧版本时，回滚

## 六、推荐配置

### 1. 高校通知公告站

推荐配置：

- 抓取方式：`请求优先`
- 字段：`title / url / date`
- 正文策略：`详情页优先`
- 分页：无或 `下一页链接`
- 最大抓取条数：`10`

### 2. 政府公告站

推荐配置：

- 抓取方式：`请求优先`
- 字段：`title / url / date`
- 正文策略：`详情页优先`
- 分页：`下一页链接` 或 `页码模板`
- 最大抓取条数：`10`

### 3. 前端渲染新闻站

推荐配置：

- 抓取方式：`浏览器优先`
- 字段：`title / url / date`
- 正文策略：`详情页优先`
- 如需登录，再补 Cookie

### 4. 接口型列表页

推荐配置：

- 请求方法：`POST`
- 补请求体
- 补请求头
- 抓取方式：优先 `请求优先`

## 七、验证方法

### 1. 手工验证

推荐按以下顺序验证一条新规则：

1. 新建规则
2. 跑 AI 分析
3. 检查站点画像与模板推荐是否合理
4. 检查列表样本
5. 检查详情预览
6. 如有问题，手动微调并重新测试
7. 保存规则
8. 执行一键复测
9. 触发一次实际更新
10. 确认文章真实入库并能在列表打开详情

### 通过标准

一条规则至少应满足：

- 列表可稳定抽到标题和链接
- 详情可稳定抽到正文或合理回退
- 正式复测不报错
- 健康状态不是连续失败

### 2. 自动化验证

当前建议的核心回归命令：

```bash
python3 -m unittest \
  tests.test_html_template_library \
  tests.test_html_detail_preview \
  tests.test_api_custom_rule_metadata \
  tests.test_custom_rule_health \
  tests.test_browser_render_fallback \
  tests.test_dynamic_spider_health \
  tests.test_article_processor_detail_strategy \
  tests.test_scheduler_incremental_guardrails
```

这些测试主要覆盖：

- 模板库与站点画像
- 详情预览
- API 元数据返回
- 健康状态回写
- 浏览器兜底
- 动态爬虫抓取
- 详情策略
- 调度增量保护

## 八、最佳实践

- 第一次建规则时，字段尽量少，先稳住 `title / url / date`
- 优先让列表样本正确，再去抠详情选择器
- 每次只改一两项，再测试
- 正式运行时，`常规追踪最大翻页数` 尽量保守
- 对需要登录的站点，先保留一份可复用的 Cookie 方案
- 对重要来源，保存成功后立刻做一次一键复测
- 如果当前规则还能工作，不要为了“更完美”频繁重生成
