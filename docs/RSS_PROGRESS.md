# RSS 功能进度记录

更新时间：2026-03-29

## 文档目的

本文档用于记录 MicroFlow 项目中 RSS 功能的当前实现进度，作为后续持续迭代时的阶段性留档。

这份记录主要基于两部分信息整理：

- 既有的 RSS 设计清单与阶段性判断
- 当前仓库代码的实际实现状态

后续每当 RSS 功能出现较大更新时，应优先修改本文件，而不是重新散落记录在聊天或临时笔记中。

## 本次稳定性修复

- 规则弹窗中的 RSS 预览现在支持真正的“取消即失效”。
- 当用户在预览过程中关闭 / 取消弹窗时，前端会立刻清除上一轮请求态，旧请求即使稍后返回，也不会再覆盖新的弹窗状态。
- 后端为 RSS 预览补充了独立取消事件，AI 排版 / 摘要阶段会尽快响应取消，不再让上一轮预览长时间占住后续操作。
- AI 请求超时策略已从固定短超时改为按内容长度和任务类型动态放宽，长 RSS 正文会拿到更长的等待窗口。
- 连续超时后的提示语也做了修正：优先提示“长内容仍未完成”，不再过早误报为单纯的 AI 服务不可用。

## 当前结论

RSS 功能已经从“方案设计阶段”进入“主链路基本成型、体验持续打磨”的阶段。

当前更准确的定位是：

- RSS 核心链路已可用
- 后端标准化能力已基本成型
- 前端规则配置、预览和阅读模式已落地
- 目前主要缺口集中在阅读器细节、图片降级、长文体验和源级高级策略

如果用一句话概括当前状态：

> 已经完成了“先结构化、再 Markdown 化、再 AI 增强、最后前端展示”的主流程，但还没有完全做到“成熟 RSS 阅读器级别的体验”。

## 阶段判断

### 整体状态

| 模块 | 当前状态 | 说明 |
|------|------|------|
| 订阅层 | 基本完成 | RSS/Atom 接入、规则保存、启停、预览、最大抓取条数、AI 双开关已具备 |
| 内容标准化层 | 基本完成 | 已实现标准化、Markdown 化、块结构、图片分类、链接补全、内容补全 |
| AI 优化层 | 基本完成 | 已拆分排版提示词与总结提示词，支持源级覆盖与回溯 |
| 展示层 | 主体完成 | 已有预览、三种阅读模式、图集展示、标签分层；阅读器细节仍待增强 |
| 测试 | 基础完成 | 已有 RSS 样例 feed 测试，但测试体系仍较轻量 |

### 当前项目定位

- 不是“刚开始做 RSS”
- 也不是“已经完全收尾”
- 更接近“RSS MVP 已完成，正在往更稳定、更像产品的方向收敛”

## 进度明细

## 1. 订阅层

### 已完成

- 支持任意标准 RSS/Atom 订阅链接。
- RSS 规则已纳入统一规则体系，保留以下核心字段：
  - `rule_id`
  - `task_id`
  - `task_name`
  - `task_purpose`
  - `url`
  - `source_type`
  - `max_items`
- 已支持源级 AI 双开关：
  - 是否启用 AI 排版增强
  - 是否启用 AI 摘要 / 彩色标签
- 已支持源级双提示词：
  - 排版提示词
  - 总结提示词
- RSS 规则支持预览后保存。
- RSS 规则支持启用 / 停用 / 删除。
- 编辑已有 RSS 规则时，保存动作现在会复用同一条验证链路，先校验 feed 再落盘。

### 相比旧记录的变化

- 这部分已不再是“方案设计”，而是实际功能已落地。
- 前端规则编辑页和后端保存逻辑已形成闭环。

### 仍可继续优化

- 订阅源健康检查还未形成完整机制。
- 源级模板库还未建立。

## 2. 内容标准化层

### 已完成

- 已实现 RSS 标准解析入口，兼容 RSS 2.0、RSS 1.0、Atom。
- 当前处理流程已明确为：
  1. 解析 feed
  2. 提取正文 HTML / 摘要
  3. 做确定性标准化
  4. 转换为 Markdown
  5. 视规则决定是否交给 AI 做增强
- 已实现块级结构拆分，当前可识别的内容块包括：
  - 标题
  - 段落
  - 列表
  - 引用
  - 图片
  - 链接
  - 分隔线
- 已实现 HTML 清洗后再转 Markdown。
- 已实现链接补全，可将相对链接转为绝对链接。
- 已实现发布日期统一处理，兼容多种 RSS 时间字段。
- 已具备内容过薄时回源详情页补正文的能力。

### 相比旧记录的变化

- 旧记录里提到“还没有独立块级 AST/JSON 结构”，现在应升级为“已完成基础版块结构”。
- 当前不仅有 Markdown 文本，还会生成并持久化 `content_blocks`。

### 图片分类状态

当前已经可以明确区分并持久化以下图片类别：

- `cover`：封面图
- `body`：正文图
- `attachment`：附件图
- `external`：外链图

这比旧记录中“图片分类尚未完全落地”的判断更进一步。

### 仍可继续优化

- 目前的块结构是“够用的基础版”，但还不是更细粒度的富文本 AST。
- 对纯文本裸链的专门清洗仍不算完整。
- 图片 alt 文本的可读性提升还不充分。

## 3. AI 优化层

### 已完成

- 已将 RSS AI 能力拆为两个阶段：
  - 排版增强
  - 摘要 / 彩色标签
- 已提供默认排版提示词和默认总结提示词。
- 已支持用户为单个 RSS 源覆盖默认提示词。
- 已支持源级 AI 开关与提示词联动。
- 当规则填写提示词时，会自动带起对应能力的启用状态。
- 手动“重新生成总结”时，已优先回溯并使用 RSS 源自身配置的提示词。
- AI 标签当前已限制为最多 3 个，每个不超过 15 字。
- RSS 手动重生成链路现在会尊重源级 AI 开关，可分别处理：
  - 仅排版增强
  - 仅摘要 / 标签
  - 排版 + 摘要
  - 清空已过期的旧 AI 结果
- 前端已兼容旧版 `summary` 兼容字段，历史 RSS 文章即使缺少独立 `ai_summary`，也能继续显示摘要模式、标签和摘要编辑内容。

### 相比旧记录的变化

- 旧记录中“排版提示词 + 总结提示词”的设想，现在已经落地。
- 旧记录中“提示词回溯”这一项，现在也已经完成。
- 现在手动重生成不再把“开启排版”误判成“必须同时重做摘要”。

### 部分完成

- 标签“按重要性排序”目前主要依赖提示词约束和返回顺序，没有独立排序机制。
- 针对“长文 / 短讯 / 图文源”的默认策略分流还未显式做出来。

### 仍可继续优化

- 增加源类型默认模板策略。
- 为图文源补“图片说明生成”能力。
- 减少重复总结成本，增强缓存和增量策略。

  - 订阅源配置

## 本轮补充记录

本轮继续做的是“收尾型稳定性修复”，没有新增 RSS 功能，重点是把现有阅读链路再压实一层：

- RSS 详情页默认阅读模式不再固定落到“增强”，而是会根据当前文章是否已有独立摘要，自动选择“摘要”或“增强”。
- 手动保存摘要、重新生成摘要、重新打开详情页这几条路径，现在会共享同一套默认模式判断，减少模式来回跳动。
- 后台静默刷新列表时不再改动全局 RSS 阅读模式，避免用户正在阅读另一篇 RSS 时被悄悄重置模式。
- 已收敛 OpenAI SDK 与应用层的双重重试，AI 失败时不再被 SDK 隐式放大，错误提示也会更明确地指向认证、路径、网络、超时或余额问题。
- RSS 规则测试/预览现在固定只返回 1 条代表样本，优先缩短长文 AI 预览等待时间，避免多样本同时预览导致误判为请求失败。

这意味着：

- RSS 的“摘要 / 增强 / 原始”三种模式本身没有变化。
- 变化的是这些模式之间的状态同步已经更一致，更接近最终交付前需要的稳定形态。
